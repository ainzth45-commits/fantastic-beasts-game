# เฟส 3: ต่อยอดขายจริง Supabase (โค้ดล้วน — เริ่มได้ทันที ไม่รอ Codex)

**เป้า:** จอ TV รับยอดขายจริงของทีมหนามแดง-BETA-01 แบบ realtime — ป้ายชื่อ+รูปพนักงานจริง · ยอดนับเฉพาะช่วง "เปิดเลี้ยง" ตามกติกา
**ข้อมูลที่เช็กแล้ว:** project `crm-sale-dashboard` (`swbxecjguiohpwllvues`) · `sales(id, employee_id, amount, created_at, product_category)` RLS on · `employees(employee_id, name, photo_url, team_id, active)` → `teams` · หนามแดง = team_id 1

### Task 3.1 — ความปลอดภัยฝั่ง DB ก่อนแตะ client (สำคัญสุด: repo เป็น public)
- [ ] สำรวจ RLS policies ปัจจุบันของ `sales`/`employees`/`teams` (`execute_sql`: `select * from pg_policies where tablename in ('sales','employees','teams')`) — ดูว่า anon อ่านได้แค่ไหน dashboard TV เดิม (crm-sale-dashboard) ใช้ key แบบไหนอยู่
- [ ] 🔶DECIDE[ทางเข้าข้อมูล] เลือก 1 จาก 2:
  - **(ก) แนะนำ: สร้าง VIEW เฉพาะเกม** `game_sales_feed` (id, amount, created_at, employee ชื่อ+รูป — เฉพาะ team_id 1, เฉพาะวันนี้) + `security_invoker=false` จำกัดคอลัมน์ ไม่เปิดเบอร์ลูกค้า/ข้อมูลอื่น · anon ได้ SELECT เฉพาะ view นี้ → client ง่าย ปลอดภัย
  - (ข) เปิด RLS SELECT ตรงบน `sales`+`employees` ให้ anon — ง่ายกว่าแต่เปิดข้อมูลกว้างเกิน (มี photo_url ทุกทีม ฯลฯ)
- [ ] Realtime: เปิด publication ให้ INSERT ของ `sales` (หรือใช้ Broadcast/Realtime บน view ไม่ได้ — ต้อง subscribe table `sales` แล้ว join ฝั่ง client จาก cache รายชื่อพนักงาน) → แผนจริง: subscribe INSERT `sales` (filter ฝั่ง client ด้วย set ของ employee_id ทีม 1 ที่โหลดครั้งเดียวตอนกดเลี้ยง)
- [ ] ⚠️ อย่าใส่ service key ใน client เด็ดขาด · anon key ใส่ได้ (public by design) แต่ต้องมั่นใจ RLS/view ปิดของที่ไม่ควรเห็น
- [ ] เทสด้วย `curl` REST + anon key จริง: อ่านได้เฉพาะที่ตั้งใจ · อ่าน table อื่น (orders/customers) ต้อง**ถูกปฏิเสธ**

### Task 3.2 — client: Supabase feed (แทน/คู่กับ mock)
**Files:** `app/src/data/supabaseFeed.ts` (ใหม่), `app/src/state/useGameStore.tsx`, `.env` → `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (anon เท่านั้น — commit ได้ แต่ใส่ผ่าน GitHub Actions secret ก็ได้เพื่อความเนียน 🔶DECIDE[ฝัง key ใน repo หรือ inject ตอน build])
- [ ] `npm i @supabase/supabase-js`
- [ ] `startFeeding()` → โหลดรายชื่อพนักงานทีม 1 (id,name,photo_url) เก็บ map → subscribe INSERT `sales` → แปลงเป็น `SaleEvent{ id: String(sale.id), amount, at, employeeName, employeePhotoUrl }` → `feed()`
- [ ] `pauseFeeding()` → unsubscribe (ยอดช่วงพักไม่นับ — กติกาเจ้านาย ไม่มี catch-up)
- [ ] กันซ้ำ: เก็บ `Set<saleId>` ที่นับแล้ว (persist) — reconnect แล้ว event เดิมมาอีกต้องไม่นับซ้ำ
- [ ] ยอดถูกลบ/แก้ระหว่างเปิดเลี้ยง: subscribe UPDATE/DELETE ด้วย — ถ้า id อยู่ใน set ที่นับแล้ว: DELETE → หักแต้มเท่าที่เคยให้ (ต้องเก็บ pointsGiven ต่อ id) · UPDATE amount → ปรับส่วนต่าง · id ที่ไม่เคยนับ → เมิน
- [ ] จอเปิดทั้งวัน: auto-reconnect (`supabase.channel` มี retry ในตัว แต่ต้องตรวจ `CHANNEL_ERROR` → re-subscribe + แสดง badge "หลุดการเชื่อมต่อ" เล็กๆ บนจอ)
- [ ] DevPanel เดิมคงไว้ (โหมด mock ยังใช้เดโม/เทสได้ — สลับด้วย query `?mock=1`)
- [ ] เทส: pure helpers (แปลง row→SaleEvent, dedupe, หักแต้ม) — เทส subscribe จริงทำด้วยมือบน dev
